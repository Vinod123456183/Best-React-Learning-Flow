### Introduction to React: The Power of Declarative UI ⚛️

React is a JavaScript library for building user interfaces, and its superpower is **declarative UI**.

#### What does “declarative” mean?

Instead of telling the browser *how* to update the UI step by step (imperative style), you simply describe **what the UI should look like for a given state**—and React takes care of the rest.

Think of it like this:

* **Imperative UI**: “Create a button. Add text. If clicked, change the text. Update the DOM.”
* **Declarative UI (React)**: “When `clicked` is true, show this. Otherwise, show that.”

You focus on **logic and state**, not DOM manipulation.

#### A tiny example

```jsx
function Greeting({ isLoggedIn }) {
  return (
    <h1>
      {isLoggedIn ? "Welcome back!" : "Please sign in"}
    </h1>
  );
}
```

You’re not manually changing text on the page. You’re declaring:

> “This is how the UI should look based on `isLoggedIn`.”

React automatically updates the UI when the data changes.

#### Why declarative UI is powerful

* **Predictable** – Same state always produces the same UI
* **Easier to reason about** – Less mental overhead than tracking DOM changes
* **Fewer bugs** – React handles efficient updates via the Virtual DOM
* **Highly reusable** – Components encapsulate both logic and UI

#### Key ideas behind React’s declarative approach

* **Components**: Small, reusable UI pieces
* **State**: Data that changes over time
* **Props**: Data passed into components
* **One-way data flow**: State → UI (not the other way around)

#### Big picture takeaway

React lets you **think in terms of UI states, not UI steps**.
You describe *what* the interface should be, and React figures out *how* to keep it in sync.

### Understanding the Virtual DOM & Reconciliation Process ⚛️

React’s performance magic comes from two closely related ideas: the **Virtual DOM** and **Reconciliation**. Together, they make UI updates fast, efficient, and predictable.

---

### What is the Virtual DOM?

The **Virtual DOM** is a lightweight, in-memory representation of the real browser DOM.

Instead of updating the actual DOM directly (which is slow), React:

1. Builds a Virtual DOM tree from your components
2. Updates this Virtual DOM when state or props change
3. Compares it with the previous version

Think of it as a **blueprint** React uses before touching the real UI.

---

### Why not update the real DOM directly?

The real DOM is expensive because:

* Layout recalculations are slow
* Repaints and reflows cost performance
* Frequent direct updates lead to janky UIs

React minimizes this cost by batching and optimizing updates.

---

### What is Reconciliation?

**Reconciliation** is the process React uses to figure out **what changed** between the old and new Virtual DOM trees.

Steps:

1. State or props change
2. A new Virtual DOM tree is created
3. React **diffs** the new tree against the previous one
4. React calculates the **minimum set of changes**
5. Only those changes are applied to the real DOM

This diffing process is fast and smart—not a full tree comparison.

---

### How React makes reconciliation efficient

React uses a few key assumptions:

* Elements of different types produce different trees
* Components with the same type will generate similar trees
* **Keys** help identify which list items changed, were added, or removed

Example with keys:

```jsx
items.map(item => (
  <li key={item.id}>{item.name}</li>
))
```

Keys help React avoid unnecessary DOM updates and bugs.

---

### Virtual DOM vs Shadow DOM

Quick clarification (they’re often confused):

* **Virtual DOM** → React concept, lives in JavaScript
* **Shadow DOM** → Browser standard, used for style & DOM encapsulation

They solve different problems.

---

### Why this matters for developers

* You write simple, declarative code
* React handles performance optimizations
* UI stays in sync with application state
* You get smooth updates even in complex apps

---

### Big picture takeaway

The **Virtual DOM** lets React simulate UI changes cheaply, and **reconciliation** ensures only the smallest, necessary updates hit the real DOM.

In short:

> React updates *smartly*, not *often*.

### JavaScript Code Execution & the Global Execution Context 🧠⚙️

To truly understand JavaScript, you need to know **how the JS engine runs your code** behind the scenes. Everything starts with the **Execution Context**, especially the **Global Execution Context (GEC)**.

---

## What is an Execution Context?

An **Execution Context** is an environment where JavaScript code is evaluated and executed.

Each context contains:

1. **Memory Creation Phase (Hoisting)**
2. **Code Execution Phase**

There are three types:

* **Global Execution Context**
* **Function Execution Context**
* **Eval Execution Context** (rarely used)

---

## Global Execution Context (GEC)

The **Global Execution Context** is created when a JavaScript program starts.

Key things it does:

* Creates the **global object**

  * `window` (browser)
  * `global` (Node.js)
* Creates `this`
* Allocates memory for variables & functions

There is **only one GEC** per program.

---

## Phase 1: Memory Creation (Hoisting)

Before executing any code, JavaScript scans the entire file.

What happens:

* `var` → initialized as `undefined`
* `let` & `const` → allocated but uninitialized (TDZ)
* Functions → stored fully in memory

Example:

```js
console.log(a); // undefined
console.log(b); // ReferenceError

var a = 10;
let b = 20;

function foo() {
  console.log("Hello");
}
```

---

## Phase 2: Code Execution

Now JavaScript runs the code **line by line**.

* Values are assigned
* Functions are executed
* New execution contexts are created for function calls

---

## Call Stack & Execution Contexts

JavaScript uses a **Call Stack** to manage execution contexts.

Example:

```js
function one() {
  two();
}
function two() {
  console.log("Hi");
}
one();
```

Call stack flow:

```
Global Execution Context
→ one()
→ two()
→ pop two()
→ pop one()
```

Last in, first out (LIFO).

---

## this in the Global Context

* Browser: `this === window`
* Node.js: `this !== global` (module-scoped)

Important interview favorite ⚠️

---

## Key Takeaways

* JS runs in **execution contexts**
* Global Execution Context is created first
* Code runs in **two phases**
* Functions create their own execution contexts
* Call stack manages execution order

---

## Mental Model to Remember

> JavaScript first **prepares** the code (memory phase),
> then **executes** it (execution phase).

Once you get this, concepts like **hoisting, closures, and async behavior** suddenly make sense.

### Event Loop, Call Stack & Concurrency in JavaScript 🔄🧠

JavaScript is **single-threaded**, yet it handles timers, APIs, and user interactions *concurrently*. The secret sauce is the **Call Stack + Web APIs + Event Loop**.

Let’s break it down clearly.

---

## 1. Call Stack 📚

The **Call Stack** is where JavaScript keeps track of function execution.

* Follows **LIFO** (Last In, First Out)
* Executes one thing at a time

Example:

```js
function first() {
  second();
}
function second() {
  console.log("Hello");
}
first();
```

Stack flow:

```
Global()
→ first()
→ second()
→ pop second()
→ pop first()
```

If the stack is blocked → the UI freezes.

---

## 2. Web APIs / Node APIs 🌐

Async operations are **not handled by the Call Stack**.

Examples:

* `setTimeout`
* DOM events
* `fetch`
* Promises (handled partly by the engine)

These are handled by:

* **Web APIs** (browser)
* **Node APIs** (Node.js)

They run in the background while the stack continues.

---

## 3. Callback Queue (Task Queue) 📥

When an async task finishes:

* Its callback is pushed into the **Callback Queue**

Example:

```js
setTimeout(() => {
  console.log("Timer done");
}, 0);
```

Even with `0ms`, it **waits** until the stack is empty.

---

## 4. Microtask Queue ⚡

Some async tasks go into the **Microtask Queue**, which has **higher priority**.

Examples:

* `Promise.then`
* `catch`
* `finally`
* `queueMicrotask`

Microtasks are executed:

> **Before** the callback queue
> **After** the call stack is empty

---

## 5. Event Loop 🔁 (The Orchestrator)

The **Event Loop** constantly checks:

1. Is the Call Stack empty?
2. If yes → run all Microtasks
3. Then → take one task from the Callback Queue
4. Repeat forever

---

## Execution Order Example

```js
console.log("Start");

setTimeout(() => console.log("Timeout"), 0);

Promise.resolve().then(() => console.log("Promise"));

console.log("End");
```

Output:

```
Start
End
Promise
Timeout
```

Why?

* Sync code first
* Microtasks (Promise)
* Macrotasks (setTimeout)

---

## Concurrency in JavaScript 🤹‍♂️

Even though JS is single-threaded:

* Async tasks run **in parallel** via APIs
* Results are queued
* Event Loop schedules execution

This is called **non-blocking concurrency**.

---

## Common Pitfalls ❌

* Blocking the call stack with long loops
* Assuming `setTimeout(fn, 0)` runs immediately
* Forgetting microtask priority

---

## Mental Model 🧩

> **Call Stack** → executes code
> **Web APIs** → handle async work
> **Queues** → hold callbacks
> **Event Loop** → decides *when* callbacks run

---

## Big Picture Takeaway

JavaScript doesn’t do multiple things at once—it **coordinates** them efficiently.

Once you master this, async/await, Promises, and performance tuning all make sense.

### Hoisting Explained: Variables & Functions 🚀🧠

**Hoisting** is one of those JavaScript concepts that feels weird—until you understand *how JS executes code*. Let’s make it simple and precise.

---

## What is Hoisting?

Hoisting is JavaScript’s behavior of **moving declarations to the top of their scope during the memory creation phase** of execution.

⚠️ Important:
**Only declarations are hoisted, not initializations.**

---

## Hoisting with `var`

`var` declarations are hoisted and initialized with `undefined`.

```js
console.log(a); // undefined
var a = 10;
```

How JS sees it:

```js
var a;
console.log(a);
a = 10;
```

✔️ No error, but value isn’t available yet.

---

## Hoisting with `let` & `const`

They are hoisted but **not initialized**.

Accessing them before declaration causes a **ReferenceError** due to the **Temporal Dead Zone (TDZ)**.

```js
console.log(b); // ReferenceError
let b = 20;
```

TDZ exists from:

* Start of scope
* Until the variable is declared

---

## Hoisting with Functions

### Function Declarations ✅

Fully hoisted—can be called before definition.

```js
sayHi(); // "Hi"

function sayHi() {
  console.log("Hi");
}
```

---

### Function Expressions ❌

Behave like variables.

```js
sayHello(); // TypeError

var sayHello = function () {
  console.log("Hello");
};
```

Why?

* `sayHello` is hoisted as `undefined`
* Calling `undefined()` throws an error

---

### Arrow Functions ❌

Same hoisting behavior as function expressions.

```js
greet(); // ReferenceError or TypeError

const greet = () => {
  console.log("Hello");
};
```

---

## Summary Table 📊

| Type                 | Hoisted               | Initialized | Callable Before |
| -------------------- | --------------------- | ----------- | --------------- |
| `var`                | Yes                   | `undefined` | ❌               |
| `let`                | Yes                   | ❌ (TDZ)     | ❌               |
| `const`              | Yes                   | ❌ (TDZ)     | ❌               |
| Function declaration | Yes                   | Yes         | ✅               |
| Function expression  | Yes (var rules)       | ❌           | ❌               |
| Arrow function       | Yes (let/const rules) | ❌           | ❌               |

---

## Common Interview Traps 😈

```js
console.log(typeof foo);
var foo = "Hello";
```

✔️ Output: `"undefined"`

```js
console.log(typeof bar);
let bar = "Hi";
```

❌ ReferenceError (TDZ!)

---

## Mental Model to Remember

> JavaScript **scans first**, **allocates memory**, then **executes line by line**.

Hoisting is not magic—it’s just **how execution contexts work**.

---

## Big Picture Takeaway

* Hoisting affects **how and when** you can access variables & functions
* Prefer `let` and `const` for safer, predictable code
* Function declarations are hoisted completely

# **2️⃣ Console Panel**

The **Console** is your **JavaScript playground**.

**Features:**

* Log values (`console.log`) for debugging.
* Execute JavaScript in the page context.
* Catch errors and warnings.
* Inspect objects interactively.

**Advanced Tips:**

```js
console.table(users);       // Display arrays/objects as a table
console.time("fetch");      // Measure execution time
console.timeEnd("fetch");
```

**Example Use Case:**

* Check why your `useState` isn’t updating correctly by logging the state:

```tsx
console.log("Current state:", state);
```

# **3️⃣ Network Panel**

* Inspect **XHR / Fetch / GraphQL requests**.

* Watch **API requests triggered by React Query or Redux Toolkit Query**.
* Inspect request payload and response to debug data fetching.

* You click a “Load Users” button.
